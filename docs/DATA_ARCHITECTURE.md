# 数据分离架构

Idle Rift 将可调整的玩法数值、可存档的运行状态、游戏规则和界面表现分为四层。

```text
data/gameplay/*.json
        ↓
GameDataRepository（加载、校验、查询）
        ↓
CombatSimulation / EquipmentGenerator（规则）
        ↓
CombatState（可存档状态） → Main UI（只展示与发送操作）
```

## 1. 静态玩法数据

`data/gameplay/` 存放不随单局变化、需要频繁平衡的配置：

- `classes.json`：职业名称、资源和职业机制参数。
- `combat.json`：英雄基础值、敌人伤害、裂隙成长和奖励概率。
- `enemies.json`：敌人身份和轮换顺序。
- `equipment.json`：部位、品质权重、基础属性、词缀池和等级缩放。

数据文件不能引用场景节点，也不能包含运行时对象。修改数值后必须运行测试，JSON 仍需通过语法和仓库校验。

## 2. 数据访问层

`GameDataRepository` 是玩法数据的唯一入口，负责读取、校验和提供查询。战斗与装备生成器通过构造参数接收仓库，因此测试可以传入受控数据源，未来也可以增加版本迁移或开发者热重载。

UI 不直接读取 JSON。新增配置查询时先扩展仓库接口，避免文件路径和原始字典结构扩散到整个项目。

## 3. 运行状态层

`CombatState` 只保存会随游戏进度改变的数据，例如金币、击杀数、职业资源、当前敌人、背包和已装备物品。它不负责攻击、掉落或界面刷新。

`snapshot()` 输出带 `schema_version` 的基础存档快照。装备完整序列化与版本迁移将在本地存档里程碑补齐。

## 4. 规则与表现层

- `CombatSimulation` 消费配置并修改 `CombatState`，不访问任何 UI 节点。
- `EquipmentGenerator` 只根据装备配置生成物品。
- `main.gd` 只读取模拟结果、刷新界面并转发玩家操作，不决定掉率或成长公式。

## 变更规则

1. 数值平衡优先修改 `data/gameplay/`，不在逻辑脚本新增魔法数字。
2. 新玩法先定义数据合同，再实现规则消费者。
3. 新的可持久化字段必须进入 `CombatState`，并考虑存档版本迁移。
4. UI 文案若代表真实数值，应来自模拟或状态，不复制另一份配置。
